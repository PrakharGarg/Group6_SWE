About

Pursuit of Hoppiness is a web app built using the Flask framework to provide an IMDB style database for beers, breweries, venues, and locations. Our mission is to make ordering a pint less stressful, to help support local breweries, and to teach people everywhere how to be beer snobs.

For the longest time, beer never had the support that wines and liquors did. Beer was beer. Now, with the boom of local breweries, microbreweries, and the craft beer scene, beer interest is in the all time high.  As fellow beer lovers, we know how much it sucks to find that perfect beer, and with this recent boom of craft beers it’s never been harder. Between the hundreds of styles, breweries, and ingredients, ordering off the tap list is now as intensive as choosing your main course. We want to help fix this.

By scraping two different databases, we have collected information on hundreds of beers (next phase!), breweries, venues, and all 50 states so that you have the best chance to order something you’ll enjoy. By showing a grid of information with the ability to see more information in something you are interested in, you can easily find the nearest place to order your new-found beer.

Our end-game it to make a fully self-sufficient web app that not only allows people to learn more about all things beer, but also use our data to make their own projects to showcase their love for beer. Through API calls to our database (next phase!), people will be able to access our information and leverage everything we have scraped. Who knows, maybe one day everyone will enjoy sipping a cool, refreshing beer!

Cheers!

Data Models

Appropriate data models follow directly from the top level design entities: beers, those tasty beverages we love so much, breweries, those hallowed sites of beer production, venues, those bars, restaurants, and pubs where we consume and convene, and the larger locales, which we decided to be US states. Each of these entities becomes a model in our design. Each has at least five attributes, a few methods which will help facilitate in their proper usage, and at least two connections to other models.

        Beers have a name, an image of their packing, a style, a flag labeling whether or not it is organic, and a rating. Beers also have a link to the brewery which produces them. Methods on beer model objects allow clients to create beers, get a textual representation of them, change the label of the beer to reflect changes to its packaging, and alter the rating of the beer.

Breweries have a name, an image of their logos, the type of brewery it is, the address, and the date the brewery was founded. Breweries link to the beers they produce, the venues which dispense their beers, and the state in  which they preside. Methods on brewery objects allow creation of breweries, retrieval of a brewery’s textual representation, altering of a brewery’s name, changing the founding date, and addition and removal of venues to which the brewery distributes its products to.

Venues are the place where we get together to enjoy each other’s company, to make new friends, and, of course, to drink beer! Venues have a name, images showcasing their ambience, signage, layout, et cetera, their address, a category (for example, is this venue a pub? Is it a bar?), and a flag indicating whether or not this venue is open to the public. Venues connect to the beers they serve, the breweries where those beers are made, and the state in which the venue is in. In order to maintain, manipulate, and properly access venue objects, several methods have been provided: one creates new venues, one gives a textual representation of a venue, one adds a beer to the list of ones it serves, another removes a beer from this list, and, finally, one method allows for the removal of every beer from a given brewery.

State are the larger locale which groups together all the different components necessary to fully represent the wonderful world of beer. States have their state name, their capital city, the postal code, or abbreviation, for the state, the official state flower decreed by the state’s government, and an array of images which show iconic sites, scenes, and cities of the state. States link to breweries and venues which are located in the state. To manage and query states, methods to create new states (in the database! Not in the real world), get textual representations of states, determine whether a state is the best one in all of the US, get the images for this state, and to determine whether or not the state flower is pretty.

API 

        To retrieve the data to populate the grids and individual pages for each beer, venue, and brewery we used two APIs, Untappd’s API, and BreweryDB. The only data we retrieved from BreweryDB was the isOrganic attribute and the year that each brewery was founded. All other attributes (refer to Data Models above) were pulled from Untappd.

        Both Untappd and BreweryDB return JSON formatted responses from each request. It’s important to note that both Untappd and BreweryDB have a limit on requests. For Untappd requests are limited to 100 per hour, while BreweryDB limits requests to 400 per day (unless you purchase a premium membership for unlimited requests). For Untappd, if you wish to have access to their API you should expect to have a potential wait of several weeks before they will approve your request and send you your client ID and client secret keys. BreweryDB has no wait and you should receive your access key by email immediately.

        For more information on how to connect to the APIs click the links below to read the documentation for each API:

        Untappd: https://untappd.com/api/docs

        BreweryDB: http://www.brewerydb.com/developers/docs 

Our Apiary page provides an easy to understand layout, showcasing greater detail regarding as to how exactly our API functions, including the different requests users can make as well as the responses for the different models in our API. These requests include: “List of all Beers”, “Details of a Specific Beer”, “List of all Breweries”, “Details of a Specific Brewery”, “List of all Venues”, “Details of a Specific Venue”, “List of all States”, and “Details of a Specific State”.

Tools

Front-End

  • Twitter Bootstrap - we used bootstrap to make a standardized front-end that would react well to dynamic screen sizes.
  • Carousel - we used carousel to display a cycle of photos on our splash page.
  • Grid System -  using bootstraps column system, creating the pages for beers, breweries, venues, and states was a lot easier than doing it by hand. Bootstrap by defaults splits the page into 12 columns, so if you wanted 3 items next to each other, you would add the class “col-3-md” to each of the items.

Back-End 

  • Flask
  • Flask is Python’s web development framework that makes it easy to construct an app in little time. By combining a few route methods to html files, one can get a flask app in a few minutes. Furthermore, flask allows one to use Jinja templates for html. Jinja allows developers to embed Python blocks into the html. Jinja also allows one to have html files inherit from other html files. For our purposes, we created a base.html file that outlines the navbar, stylesheets, and title info. All of our other html files, inherit from base, and let us carryover similar elements.

Makefile 

The makefile currently only has recipes for check (the default), clean, run, deploy, doc, env, and test. Check makes sure that the files included in the FILES variable exist. Clean removes the .pyc, __pycache__, virtual environment, IDB1.html, and IDB1.log from our project directory. Run simply starts the python interpreter with main.py as its single argument. Deploy will start a new deployment of the application with the contents of the callee's location machine. Doc runs pydoc on our models.py file and renames the output to IDB1.html. It also runs git.log and stores the output in IDB1.log. Env tries to install virtualenv, creates a new virtualenv, activates it, and then installs the dependencies in requirements.txt. Test currently does nothing because we are not required to run our unit tests.

Travis

In the .travis.yml file, we set the language to python and the python version to 3.5. The script rule only calls date, ls -alr, and make check. So our travis build will pass if all of the files we are expecting to see are present.

Hosting

Google App Engine Flex

We used the Google Cloud Platform to host our website. There are quite a few hosting options on the Google Cloud Platform. We chose to go with the Google App Engine. The Google App Engine has two environments that you can deploy your applications to: the standard and the flexible. We decided to use the flexible because it allows you to have more control over the Compute Engine instances while still not having to deal with the hosting environment. For our project, the only dependency requirements are Flask-0.12, Flask-SQLAlchemy-2.0, and gunicorn-19.7.0. Gunicorn is a python http server that is compatible with quite a few web frameworks, including flask. Another feature of the flexible environment is that it can build our dependencies by reading from a file called requirements.txt. So, if you use a python virtual environment to test your application locally, you can run the command

        $ pip freeze > requirements.txt

and have the same dependencies installed on the App Engine instance.

        We did the deployment from the command line using the Google Cloud SDK. There are instructions on the website for how to install it and add gcloud to your environment path. This will include the tool gcloud which is the primary command-line interface to the Google Cloud Platform.

        Deploying the application requires that you have an App Engine project under the Google Cloud Platform account that is going to deploy the website. You can do this manually on the Google Cloud Platform console. You also need a .yaml file in the root directory of your project. Ours is called app.yaml. The file specifies how the instance will be built. We specify that we want to use python, the flex environment, and the entrypoint, which tells the App Engine how to start the app. Our entrypoint field looks like this

        

        Entrypoint: gunicorn -b :$PORT main:app

Gunicron is used to serve our application. The -b is for binding and the $PORT variable is set by the App Engine when it starts the application. The main:app specifies that the application should start in a file named main, which in our case, is main.py. The runtime_config and python_version specify that we want the default python3 version to be used, which is 3.5.e a

Once you have the app.yaml setup correctly, you can initialize gcloud and deploy the application. To start this, run

        

        $ gcloud init

In the root directory which has the app.yaml. This will prompt you to choose the account you would like to deploy the application under and then the project you would like to deploy to. Now you can run the command to deploy:

$ gcloud app deploy

 It will also ask you to confirm that you want to deploy your application to <project-name>.appspot.com which you should respoond with ‘Y’. The deployment takes a few minutes and takes even longer if you did something wrong it fails. Once it is finished deploying, you can navigate to <project-name>.appspot.com to see if it looks the way you expected. Alternatively, you can run the command

        $ gcloud browse

Which will open up that site in your browser.

        

Connecting Custom Domains

Connecting a custom domain name to the application requires google to verify that you own the domain, adding the custom domain to the App Engine, and adding the necessary records to the domain within the domain registrar. There is a nifty tutorial that we used that goes over a couple of different ways that you can verify you own a domain. We used the domain name provider option which requires you to add a TXT record that google provides to you. If you are using namecheap, you can add records in the advanced dns section. Once ownership has been verified by google, you can add the domain as a custom domain by navigating to the App Engine, Settings, custom domains section of the Google Cloud Platform console. You can select the domain you want from the list of verified domains. Google will then provide you with a couple of A and AAAA records to add to the advanced dns setting of your domain registrar. You also need to delete the conflicting A records that route to 192. These setting can take up to 24 hours to take affect, but you should be able to see the same thing on <project-name>.appspot.com and the domain that you configured.
